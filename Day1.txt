order book ka data hamara in memory mai store hota hai kyuki trading ko bilkul seconds mai karvana hota hai hai, agar jaise mujhe ek stock buy karna hai or phir voh jake mere stock database se match karke la rha hai, itne mai toh bahut time lag jayega or phir traffic sa create ho jayega, isliye ise in-memory store karte hai 

Difficulties kya kya hai - 
1 - Scalability (Horizontal Scaling nhi kar payenge) kyuki hum apne order books ko in-memory store kar rhe hai 
order book ki process hi alag hi hota hai mtlb main server se different hota hai or ye single thread ke way mai run hota hai or ye process hamara node.js mai ban sakta hai or rest se bhi bna sakte hai, isko ase alag se run karne se hum horizontal scaling kar sakte hai or Scalability increase kar sakte hai 
2 - Data Crashing ki dikkat hoti hai - par jaise agar data crash ho gya or order book process mai saara data chala gya toh jo queue se order book process hamara data le rha tha, usi queue se vapis restore kar lega, mtlb us queue mai hamara data secure rahega 

Hamare project ka architecture - 
hamare main server or order book ki process ko queue connect karega 
browser se toh pehle api se request karenge phir voh jaise order kar rha hai buy ya sell ka toh ek order id ban gyi hai hamare api server mai, voh order id hamara queue pe gya, phir queue se order book ka process us order ko nikal lega phir match karke jitne sell ya buy karne ka kaam hoga voh kar dega or baki cheejen jo pending rahe gyi usko phir alag queue mai rakhega, for e.g. pending stocks vagera phir jo ye queue bana jismai hamara updated data hai, us queue mai se web socket data ko uthayega or voh data phir browser mai jake show hoga, isi vajah se bhi hamen constantly updated data browser pe show hota hai. or jo ye hamara queue hai jismai updated data store tha iska data phir hum database mai bhi store karva sakte hai
